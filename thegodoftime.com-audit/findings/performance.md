# Performance Audit — Core Web Vitals & Page Load

**Site:** https://www.thegodoftime.com
**Date:** 2026-07-14
**Method:** Real Lighthouse 13.4.0 lab runs (mobile, default simulated-throttling config — Slow 4G network simulation + 4x CPU slowdown) executed live against production via `npx lighthouse` + headless Chrome 150, one run per page (single-sample, not median-of-N). Cross-checked against source code (`app/layout.tsx`, `components/clocks/experiences/WorldClock.tsx`, `public/`, `app/manifest.json`).
**Field data (CrUX) status:** NOT AVAILABLE. No Google API key is configured in this environment (`.env` has no PageSpeed/CrUX key) and the unauthenticated PSI endpoint returned HTTP 429 immediately. The site is also very new (GA added 2026-07-11), so it likely has insufficient 28-day Chrome-user traffic to appear in CrUX regardless. **All Core Web Vitals figures below are single-run lab/simulated data, not real-user field data** — treat absolute LCP seconds as directional/worst-case (simulated mobile throttling is intentionally pessimistic vs. real-world 75th-percentile field data), but treat the *relative* comparisons, TBT/long-task attribution, resource sizes, and source-level findings as high confidence since they come from actual trace data and source inspection, not estimation.

## Findings

### Finding 1: Homepage and World Clock LCP is Poor in lab testing (7.8s), driven by main-thread contention, not network
**Severity:** High
**Description:** Lighthouse measured LCP of 7.83s on both `/` and `/clocks/world-clock` (mobile, simulated throttling) — more than 3x the 2.5s "Good" threshold and past the 4.0s "Poor" cutoff. Server response (TTFB) is fine (70–194ms real, well under budget) and the LCP element itself is plain text (a `<span>` on Home, a `<p>` description on World Clock) with no image decode cost. Lighthouse's `lcp-breakdown-insight` only attributes ~2.6s to TTFB + "element render delay" — the remaining ~5.2s gap on both pages lines up with the `mainthread-work-breakdown` data: Home shows 1.6s of Script Evaluation + 1.1s of Style & Layout + 1.0s "Other" work competing on the main thread before the browser gets to paint the LCP node. In other words, the bottleneck is CPU contention from JS execution/hydration and style/layout recalculation delaying paint, not a slow server or a heavy hero image.
**Recommendation:** (1) Identify and defer/lazy-load any non-critical client component work that runs during initial hydration on Home and World Clock — profile with `React DevTools Profiler` or Chrome Performance panel to find what's consuming the 1.6s of Script Evaluation before first paint. (2) Reduce the amount of client-rendered ("use client") code in the above-the-fold path; where possible keep the LCP element (and its ancestors) in a Server Component so it paints before hydration-dependent JS finishes. (3) Since real TTFB is already good, prioritize this over server/CDN work.

### Finding 2: Age Calculator and Countdown Timer have severe Total Blocking Time (1.2–1.8s) — high INP risk
**Severity:** High
**Description:** TBT was 1,767ms on `/tools/age-calculator` and 1,222ms on `/clocks/countdown-timer-online`, both far past the ~200ms range associated with good INP. The `long-tasks` audit attributes this to (a) the page's own bundle (single tasks of 640ms and 498ms respectively — a single task over 500ms will, by definition, make any interaction landing in that window fail the 200ms INP "Good" bar and likely the 500ms "Poor" bar too), (b) a shared vendor/app chunk `3j9pm5otqxm82.js` (72KB, 37.6% unused per the `unused-javascript` audit) contributing 100–500ms tasks on both pages, and (c) the GA `gtag.js` script contributing 150–510ms of blocking time during hydration on every page tested. `mainthread-work-breakdown` confirms Script Evaluation dominates (3.5s total on Age Calculator, 1.8s on Countdown).
**Recommendation:** (1) Code-split `3j9pm5otqxm82.js` — it appears across Home, Age Calculator, and World Clock and carries ~27KB of unused code on Home alone; audit what's bundled into it (likely a shared UI/vendor chunk) and move calculator-specific logic behind dynamic `import()` so pages only load what they render. (2) Break up the large single tasks (640ms/498ms) attributed directly to the page route — these are almost certainly synchronous work done during initial render/hydration (e.g., building calculator state, formatting all outputs at once); chunk this into smaller units with `requestIdleCallback`/`scheduler.yield()` or defer non-visible work. (3) These two pages should be prioritized first for INP remediation — they are the most interaction-heavy pages in the sample (form inputs, live countdowns) and currently have the worst blocking-time profile.

### Finding 3: `icon-192.png` and `icon-512.png` are identical, mislabeled, oversized files (562KB each) fetched during page load
**Severity:** Medium
**Description:** Source inspection of `public/icon-192.png` and `public/icon-512.png` shows both files are **byte-for-byte identical** (`cmp` confirms), both are actually **1024×1024 baseline JPEG data saved with a `.png` extension**, and both are 562,015 bytes — despite `app/manifest.json` declaring them as `192x192` and `512x512` PNGs respectively. Neither file has actually been resized or re-encoded to match its declared dimensions/format. The Lighthouse network trace for the homepage confirms `icon-192.png` is fetched during the page-load trace (562KB transfer) — it was the single largest resource downloaded on the homepage, bigger than all JS chunks combined. Browsers fetch manifest icons during PWA-installability evaluation, so this cost is being paid on real page loads, not just on manual "add to home screen."
**Recommendation:** Regenerate proper icon assets: a real 192×192 PNG (or better, WebP) for `icon-192.png` (should be a few KB, not 562KB) and a real 512×512 PNG for `icon-512.png`. Also correct the underlying format — files should genuinely be PNG-encoded (or switch `manifest.json`'s `type` to `image/jpeg` if JPEG is intentional, though PNG/WebP with transparency is standard for app icons). This alone should cut well over 500KB of wasted transfer per relevant page load.

### Finding 4: World Clock's per-second tick triggers non-trivial recomputation for every pinned city — real INP risk during sustained use, not fully captured by a single Lighthouse trace
**Severity:** Medium
**Description:** Source review of `components/clocks/experiences/WorldClock.tsx` confirms the concern flagged in the task brief. The component runs `setInterval(..., 1000)` (line 399) that calls `setTick` every second, forcing a full re-render of the clock grid. On every render, `getClockData()` (lines 441–540) is called once per pinned clock (up to 10) and **instantiates two new `Intl.DateTimeFormat` objects per clock, per tick** (one for the target timezone, one for the local timezone) plus a third formatter for the GMT-offset string — none of these formatter instances are memoized/reused across ticks. `Intl.DateTimeFormat` construction is relatively expensive to instantiate repeatedly; doing this for up to 10 clocks, every second, indefinitely while the page is open, creates a recurring main-thread task that increases the odds of a user's tap/click landing inside a tick's re-render window and getting delayed — this is a classic INP-under-sustained-interaction pattern that a single ~10s Lighthouse load trace does not exercise (Lighthouse's TBT of 350ms for this page reflects only the *load* phase, not ongoing tick cost). Additionally, when the "BOARD" (split-flap) display mode is active, `playSplitFlapClick()` (line 121) creates a **brand-new `AudioContext` on every single flipped character** rather than reusing one instance — `AudioContext` construction is a comparatively heavy synchronous browser API call, and this fires for every character of every clock that changes each tick. A separate `setInterval(handler, 400)` (line 300) also polls `document.fullscreenElement` every 400ms indefinitely for fullscreen-state detection instead of relying solely on the `fullscreenchange` event that's already wired up.
**Recommendation:** (1) Cache `Intl.DateTimeFormat` instances per timezone (e.g., in a `Map` keyed by resolved IANA zone name, created once and reused) instead of constructing 2–3 new formatters per clock per second. (2) Create a single shared `AudioContext` once (e.g., on mount or lazily on first user interaction, which is also required by browser autoplay policies) and reuse it for all split-flap click sounds instead of instantiating a new one per flip. (3) Remove the 400ms fullscreen-polling `setInterval` and rely on the `fullscreenchange`/`webkitfullscreenchange` listeners alone, which are already present. (4) Consider memoizing per-clock render output with `React.memo`/`useMemo` keyed on the tick-derived time string so unrelated state changes (e.g., opening the timezone dropdown) don't force recomputation of all pinned clocks. This same tick pattern is worth auditing on any other clock/countdown component (e.g., Sunrise/Sunset Calculator, as flagged in the brief) for the same anti-patterns.

### Finding 5: Google Analytics (`gtag.js`) is correctly deferred but is a meaningfully-sized, partially-unused script contributing real blocking time
**Severity:** Low–Medium
**Description:** `app/layout.tsx` loads GA via `next/script` with `strategy="afterInteractive"` (lines 81–92) — this is the **correct** loading strategy (it won't block initial render/LCP), and confirmed in the network trace it starts loading in parallel with other resources, not blocking. However, the script itself is 163.7KB transferred, and Lighthouse's `unused-javascript` audit found **41.98% (67.4KB) of it goes unused** in the trace window. It also showed up directly in the `long-tasks` audit on every single page tested, contributing single tasks of 91–510ms during hydration (worst on Countdown Timer: two GA-attributed tasks totaling ~690ms). Because `afterInteractive` scripts run once the page is interactive — i.e., around the same time a real user might be attempting their first interaction — GA execution is a plausible contributor to poor INP on first interaction, particularly stacking on top of the app's own hydration work.
**Recommendation:** The loading *strategy* is already correct and should not be changed to `beforeInteractive` or `afterInteractive`→`lazyOnload` is worth testing, since GA doesn't need to be ready in the first second of interactivity for correctness (events queue via `dataLayer.push`). Consider `strategy="lazyOnload"` or manually delaying `gtag.js` injection by 2–3s (or until `requestIdleCallback`) to move its main-thread cost outside of the highest-INP-risk initial-interaction window, at the acceptable cost of very early (sub-3s) analytics events being slightly delayed. This is a secondary priority relative to Findings 1–2.

### Finding 6: Global CSS bundle (27KB) is render-blocking on every page — moderate, not urgent
**Severity:** Low
**Description:** The `render-blocking-insight` audit flagged `_next/static/chunks/1up20co_xlas8.css` (27.1–27.2KB transferred) as render-blocking on all four pages tested, with an estimated 150–166ms of blocking duration each. A second, tiny (1.7KB) CSS chunk is also render-blocking but negligible. This is a normal Next.js global-stylesheet pattern (likely Tailwind's compiled output) and 27KB is not large by modern standards, but it is fully render-blocking rather than split into critical/non-critical.
**Recommendation:** Lower priority relative to the JS/LCP findings above, but worth revisiting: audit the global stylesheet for unused rules (Lighthouse's `unused-css-rules` audit can quantify this per page) and consider whether any of it can be scoped to specific routes/components to shrink the render-blocking payload, since this same file loads on every single page regardless of whether the page uses all its rules.

### Finding 7: Font loading strategy is solid and not a meaningful bottleneck
**Severity:** Informational (positive finding)
**Description:** `app/layout.tsx` uses `next/font/google` for all four font family variants (Cormorant Garamond, Space Grotesk ×2 weights-config, JetBrains Mono). This self-hosts the fonts from the same origin (confirmed in the network trace — all `woff2` requests are to `thegodoftime.com/_next/static/media/...`, not `fonts.googleapis.com`), eliminates the extra DNS/connection cost of a third-party font host, and `next/font` automatically applies `font-display: swap`-equivalent behavior and preloading. Lighthouse's `font-display-insight` audit found zero issues on every page tested. The 5 font files total ~134KB but load in parallel starting at ~386–388ms and all complete by ~600ms — well before they'd become a bottleneck, and CLS was 0 on 3 of 4 pages tested (see Finding 8 for the one exception), indicating no visible font-swap layout shift in practice.
**Recommendation:** No action needed. If further trimming is desired, `Space_Grotesk` is loaded twice with different weight configs (`--font-ui` and `--font-headline`) — consolidating into a single font instance with all needed weights could shave one HTTP request, but this is a very minor optimization.

### Finding 8: Age Calculator has the only non-zero CLS in the sample (0.084) — still "Good" but worth a source check
**Severity:** Low
**Description:** `/tools/age-calculator` measured CLS of 0.084 — comfortably inside the "Good" (≤0.1) band, but the only page in the four-page sample with any measurable shift at all (the other three all measured 0). This is very likely tied to the page's date-input default values, live "age in seconds" ticking output, or a validation/results panel that mounts after initial paint (common on calculator pages where the result area is conditionally rendered post-hydration).
**Recommendation:** Not urgent since it already passes the threshold, but worth a quick check with Chrome DevTools' Layout Shift Regions overlay on this specific page to confirm the source and reserve space (fixed min-height container) for the results panel if it currently mounts without pre-allocated space, to keep margin against the 0.1 threshold as content is added to the page over time.

### Finding 9: Decorative ad-rail `<aside>` elements are low-cost currently, and AdSense loader is correctly inert
**Severity:** Informational (positive finding, no action needed)
**Description:** Confirmed in `app/layout.tsx` (lines 93–100, 110–134) that the AdSense script (`adsbygoogle.js`) is wrapped in `{ADSENSE_CLIENT_ID && (...)}` and `NEXT_PUBLIC_ADSENSE_CLIENT_ID` is unset in this environment, so **the AdSense loader does not render or fire at all currently** — confirmed by its absence from every Lighthouse network-requests trace across all four pages. The two decorative `desktop-ad-rail-left`/`right` `<aside>` elements do render on every page (they are static placeholder markup, no images, no iframes, just text/emoji), and did not register as layout-shift culprits or meaningfully add to DOM size (Home's total DOM element count was 370, Age Calculator 275, World Clock 368, Countdown 242 — all comfortably under the 1,500-element "excessive DOM" threshold).
**Recommendation:** No action needed today. When AdSense is eventually enabled (once `NEXT_PUBLIC_ADSENSE_CLIENT_ID` is set), re-run this audit — real ad creative (especially auto-sized/responsive units) is one of the most common sources of new CLS regressions, and the currently-empty rail containers should have their dimensions/`min-height` locked to the eventual ad slot size *before* ads go live to avoid a shift when real ads start rendering into that space.

## What Works

- **TTFB is genuinely good across the board** (70–194ms measured via Lighthouse's `server-response-time` audit on all four pages tested) — Vercel hosting/edge delivery is not a bottleneck.
- **Font loading strategy is well-implemented**: `next/font/google` self-hosts all 4 font families from the same origin with automatic `font-display` handling and no third-party font-host round trip; zero issues flagged by Lighthouse's font-display audit on any page.
- **CLS is essentially solved**: 3 of 4 pages measured exactly 0 layout shift, and the 4th (Age Calculator) is well within the "Good" band at 0.084. No evidence of image-dimension, ad-rail, or font-swap shift problems.
- **Third-party script hygiene is good in two respects**: GA is loaded with the correct `afterInteractive` non-blocking strategy (not render-blocking), and the AdSense loader is correctly environment-gated and does not fire at all while inactive — confirmed absent from every network trace.
- **DOM size is well controlled** on all sampled pages (242–370 elements, all far under the 1,500-element risk threshold), despite these being fairly dense, interactive calculator/clock UIs.

## Confidence & Methodology Notes

- All Core Web Vitals numbers in this report are from **live Lighthouse 13.4.0 runs against production** (`https://www.thegodoftime.com`), mobile form factor, default simulated throttling, **single run per page** — not a median of multiple runs, and not real-user CrUX field data (unavailable in this environment; no API key, and the site likely has insufficient CrUX traffic volume regardless given it added GA only 3 days ago). Lab/simulated LCP numbers, especially the ~7.8s figures for Home and World Clock, should be read as a pessimistic, worst-case-network-and-CPU signal rather than a literal prediction of what most real visitors experience — but the underlying *cause* (main-thread contention from JS during hydration) is a real, source-confirmed bottleneck that will affect real users proportionally even if the field-measured seconds are lower.
- Resource sizes, long-task attribution, `unused-javascript`, DOM size, render-blocking resources, and font-display findings are all from actual Chrome trace data captured during these runs — high confidence.
- The World Clock ticking/INP-under-sustained-use finding (Finding 4) and the icon file-format bug (Finding 3) are based on direct source-code and binary-file inspection (`components/clocks/experiences/WorldClock.tsx`, `public/icon-*.png`, `app/manifest.json`), not on a Lighthouse trace — these are logically-derived risks/bugs confirmed by reading the actual code/files, not measured live-interaction INP (no field RUM tooling is installed on the site to measure this directly).
- Recommend re-running this Lighthouse suite (and, once traffic accrues, pulling real CrUX data) after implementing the Finding 1–4 fixes to confirm improvement, and specifically recommend adding a `web-vitals` library RUM beacon (reporting to GA4, which is already installed) if real-user INP/LCP/CLS monitoring over time is desired, since CrUX will take time to populate for a low-traffic new site.
